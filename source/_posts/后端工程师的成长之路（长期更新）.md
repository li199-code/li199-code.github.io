从这篇文章开始，我将参考这个网站提供的[技术路线](https://roadmap.sh/backend)进行学习。因为技术内容会专门写新文章，所以本文是记录学习进度和一些偏感性的认知，顺带工作过程中的一些感悟。

## 2023.12

- 12.21

工作中大部分时间是在找问题出在了哪儿，真的解决问题的时间少，二者大概七三开。

- 12.29

这两天做了一些非常恶心的工作，实在让人头疼，第一次对这些数据工作感到厌烦。我现在就是取数，展示，还没有理解到数据库的核心，感觉就很累了。

## 2024.1

- 1.4

这几天上班好像进入一种高原疲劳期，处理的事情繁杂细碎，工作是在别人留下的屎山上完成的。这或许就是学生向打工人的一种转变过程。就好像运动训练一样，去感受变化带来的强度，围绕它规划自己的休息和习惯，保持精力充沛。

- 1.7

周末了，想聊点最近思考的东西。

最近也算是深刻体会到了为什么很多人上学时开通了博客，但是毕业工作后就很少更新了。上班很容易耗尽注意力，进入一种心累的状态。下了班到家，就想躺床上刷手机，被动娱乐，难以主动学习。要想破局，要么是在上班时间，人的注意力还算充沛的时候，能接触思考富有锻炼性质的需求，从而提升能力（这对于程序员来说确实是自我提升的最快方式）；要么就是进行有效的精力管理，让自己的身体和脑子适应下班后保持学习的强度。

还有，我一直以来有一种疑问：我一直在 crud 的圈子里打转，尽管算上研究生开始将计算机作为吃饭的家伙已经有三年多了，但是自己似乎还是没有突破下一个层级。最近想看一些数据库的书，却发现到中间看不下去。那些并发、锁之类的概念，书中也没有具体的例子，不知道怎么用。我陷入了一种迷茫，不知道下一步学什么。当我在知乎上看到一篇回答，我得到了一些启发。

- 1.13

钱难挣，屎难吃。一份工作除了薪水，还要考虑到恶心的事情多不多，比如和领导的理念是否相和，不能的话就很难受；还有公司规章制服是否人性、加班时长、工作内容等，不赚最多的钱，而是赚性价比高的钱。实际生活中，和同事的交集限于上班的那段时间，不把上班的情绪带到下班后。每天做好精力管理，早睡早起。这就是我上了半年班总结出来的道理。

- 1.16

总结一下我这段时间上班做的东西，思考一下可以深入学习的方向，提炼一下关键字。

| 工作描述                                                                         | 关键字              |

还有自己做过的所有需求几乎都是面向后台的，比如后台管理系统，下一步是不是也要找点面向前台的需求呢？

- 1.25

这周非常有戏剧性。上周本以为充电桩项目已经基本完成了，这周三却被告知根本没入手最核心部分：将各接口按照实际流程组合起来测试一遍。结果做起来才知道坑爹之处：首先是 tcp 反直觉，tcp 报文一次发送，并不能收到对应的返回。其次，是三人联调，好处是出了问题一起想，坏处就是沟通成本高。

需求是做不完的，这几天因为各种各样的原因，休息的也不好。确实要注意精力管理了。

- 1.27

因为想尝试在 linux 上尝试安装 postgresql，我又重拾了 ubuntu 操作系统，但是发现很多命令都忘了。工作中其实也在和 linux 打交道，比如登录测试环境查看服务报错情况，不过运行的命令都很固定，一般是找到正确目录后就达到目的了。回看我博客网站上的标签列表，发现大部分标签的知识其实都很陌生了。所以，知识没有经常复习，就会遗忘。虽然说现代的开发环境中，有 google 和 gpt，但是，如果总是查一些简单的概念和问题，那就永远无法摆脱 junior 的阶段。所以，我应该对后端工程师必会的几个领域的知识做到定期复习。除了基础的 nodejs、postgresql，还有 redis、git、linux、docker 等。

## 2024.2

- 2.3

从古早时代，单机服务器即可处理大部分访问，到现在互联网巨量请求使得服务必须满足高可用性。这种变化让代码的功夫只占了一个程序员能力的一部分，更多的来自于代码之外的知识理解。

- 2.16

春节假期零散地看了《Redis 实战》这本书，这本书的内容和它的书名对应的很好，确实举的都是实例，给出了大量的代码实现。目前公司项目代码里没看到 redis 的稍微复杂的应用，都是简单当作一个高速数据库在用的。我打算整理一下 redis 最常用的一些场景，结合书来掌握。

等我学过一遍 redis、kafka 和 mongodb，并把做过的项目梳理完写到简历里，我就准备换工作了。

今天看到一个不错的讲解如何应对技术面试的文章，虽然是以 java 为例，但是具有很强的参考性：

- 2.21

昨天参加了一场面试，一如既往的感觉不好，答上来的不多。整理一下问的问题吧：

> 了解 clickhouse/mysql 吗，我说不知道 clickhouse，他就给我解释了列式数据库，速度很快。问我每天的数据量，我说两万多条订单，他说他的数据是一百多万条 😄。
> Redis 部分，持久化方式，缓存穿透、哪些数据适合存储在 mysql，然后进入缓存，哪些数据只能在 redis。
> 两道情景题，占据了绝大部分时间。第一题，看我做了个 socket 的请求和转发项目，就问我内网穿透，即内网可以访问外网，反过来不行。第二题，用户自定义排序，加上十万条数量级的条件。
> Nodejs: 用过 koa、express 吗？Nodejs 的优缺点？Nodejs 和 Javascript 的异同？
> 分布式、微服务了解吗？仅仅问了概念。
> 最后就聊了一下 python、说他们最近要拿 u2net 搞图像分割。

总结，场景题是真不会。准备的项目描述是一点没问。大悲~

- 2.25

周五参加了一个智能工厂解决方案的公司的面试，面试官问的都是一些八股文，浅浅记录一下：

> Nodejs 部分：Eggjs 框架介绍（有点无语），Buffer 和 Stream（我听成 string）,websocket/socketio、事件循环机制
> 计算机网络：TCP 和 UDP 的区别
> 项目：项目介绍、QPS 多少（不知道）、怎么用的 redis，怎么发送的报文
> 其他： restful 规范是什么

- 2.29

前天面的那家公司进二面了，本来以为手拿把抓，甚至都开始考虑杭州租房了。事实证明，半场开香槟了。今天一问，挂了。坦白说，如雷轰顶。许是自己事实上没有经历过什么大的挫折和失败。今天一天意志都消沉。明白为什么那么多人喝酒抽烟了，仅仅是麻木自己，远离尘世里的烦恼一刻，就已经非常值得了。

今天失败后得到的反馈是，经历不够丰富。虽然说很可能是一种托辞，不见得是真实原因，我的简历还是需要好好审视的。首先，经验不够，一年不到，容易被正经公司挑刺。其次，项目经历有点水？我自认真实，没有吹嘘，但是经此一事，我认为找工作五分实力，五分靠吹。项目功能说不上来，给人一种参与感不高的感觉。

至于接下来怎么做，我打算把停车里面的功能梳理一下，挑出一些亮眼的，当作是我做的。然后学学 typescript，转到快递那边，这样就算做过两个大项目了。

## 2024.3

- 3.6

这周一一口气面了两家，全是不到二十人的小公司。第一家老板就是纯不懂技术的，招聘要求写错，找的是后端还要求会前端；进去面试间也是一点技术不问，言谈间得知公司目前一个后端也没有，让我进去之后啥都得干，开发、测试、运维，薪资最多 15k。。。大哥，在上海给你干这么多的，怎么都得 20k 起步了吧。面试完果断 pass。

然后是一家做 ai 的创业公司。先让我做了两道题，一题是典型的 javascript 同步异步执行顺序题，另一条是给出一段超长的 sql，让分析做了什么。第一题顺利秒了，第二题分析了一下，是一个 CTE 语句，然后主表连接了若干个表来查询。接下来是问了几个问题：

> Redis 在项目中拿来做什么（一个是缓存，一个是锁）
> docker 查看容器的命令（docker ps）
> linux 查看进程的命令(ps)
> 会 k8s 吗（不会）
> 项目里面的高并发用到了哪些措施（分库分表，分布式节点）
> nginx 懂吗？pg 数据库是怎么实现读写分离的？（不知道，都是运维配的）
> git rebase 是什么？（不会）

剩下的就是闲聊啦，跟我介绍公司的业务，聊了聊模型结构，虽然听不懂。感觉公司还是挺 fancy 的，还是挺想去的，希望能有个好结果！

- 3.8

没想到好事来的这么快，今天我就拿到 offer 啦！能吸引我加入这家公司的最大因素，感觉还是老板。相处下来，能感受到挺舒服的，小公司的氛围怎么样，和老板本人是强关联的。希望我的直觉没错吧。毕业不到一年，这将是我第二份工作了。当初自己费劲巴拉地要跨专业考研转码，又历经千辛万苦毕业，不就是为了月薪过万吗？这一刻，我终于对自己有了一个交代。

工作经验 3 年、5 年都是一道槛，希望自己能在上海顺利积累到至少三年（还差两年多）。

## 2024.4

- 4.1

今天正式入职新公司了。怎么说呢，整个公司加老板八个人，确实不能再小了。后端开发，只有两个。第一天，就让我解决一个 clickhouse 的问题，于是花了一天的时间，学 minikube/k8s，搭环境。之前听信一个教程，不装 minikube，实际上 minikube 在 windows 上也挺好用的。

聊聊第一天上班一些不舒适的点吧。首先是和老板面对面加上试用期，还是有点心理压力的。另外午休时间相比前公司是大大缩短了，睡觉是没有的，所以真是一天干够八小时。

说完坏的再说好的。老板肯定是属于年轻人那一型的，基本没有交流障碍，能力也很强，跟着学肯定能学到很多。同事也都年轻，说话好像也都挺礼貌的。办公区域比前公司现代化了不止一点，干净的厕所，清香的泡泡洗手液，都挺好的。

- 4.6

节前的三天，最大的感受是挑战很大。这种挑战，不是那种无意义的堆工作时长，而是需要提升技能才能更好地完成任务。老板本身是技术出身，思维逻辑和工程能力很强。项目又是 AI 类，和之前做的典型互联网服务有很大的区别，需要把思维转个弯，接受围绕 AI 为核心的功能开发模式。

- 4.8

这两天时间修改一个 bug，感触良多。首先这个系统的复杂性很高，前人的代码有各种跳转。其次这次的需求包含三种情况，且代码较长。每次有一点改动，都要把三种情况测一遍。时间一长，次数一多，很容易眼花，我有多次没有把期望的例子和实际搞错了。而且，由于没有想明白逻辑，做了不少无用功。

如果说我从这两天吸取了什么经验教训的话，一是多动手记录思考过程，先在脑海里想明白大致逻辑，并记录，然后在动手过程中逐步完善，我虽然有记 markdown 笔记，但是忽略了最关键的代码逻辑，导致多做了很多无用功。二是对于这种多情况需求，应该尽早写好测试用例，这样每次改好一版就运行，而不是对着一个 debug 文件，每个场景修改一次，这种做法对人的身体和心理是一种很大的负担。

- 4.11

没入职前觉得工作内容会是运维开发工程师，从这些天的工作内容看，似乎并不需要掌握较深的运维技术。k8s 只在刚入职时，为了解决一个数据库问题时体验了一下。docker 倒是日常都在用，常用的有 mongodb、mysql、clickhouse。

随着新公司入职过渡趋向稳定，开始思考下一步的学习方向。语言方面，JavaScript 和残废的 typescript。语言的学习反馈也是最强烈的，可以深入学习 JavaScript 和 typescript。后续就是数据库知识，包括 mongodb、mysql 和 clickhouse。

- 4.15

现在看来，公司这个系统的复杂程度还是超出了我的想象。现在的情况是，出现问题了，我有时还不能快速定位到问题。描述一下工作的场景：vscode 开了十多个标签页，然后每个文件的代码很多，各部分跳转频繁。debug 也是噩梦，各种递归调用。

想想一些尽可能提高工作效率的方法：

1. 熟练掌握 vscode 快捷键。现在上班一直用着自己的笔记本，屏幕不大，进一步放大了不会用 vscode 的劣势。十几个标签页一开，再想用鼠标滑动去导航栏里找文件，实在是大海捞针。目前我在习惯的一种方法是，ctrl+p 去定位。还有一个巨长的代码文件，报错信息显示了第几行，快速定位的方法是 ctrl+p，然后冒号跟行号。诸如此类。

2. debug 的 call stack 里有详细的调用栈信息。当调用关系越来越复杂时，就要看这个。多来几次后，要把调用关系记下来。

- 4.19

又到周五了，感觉这一周过的飞快~

相比于上份工作，现在简直太舒服了。不仅挣到更多的钱，而且工作环境更好了，心情更美丽了，个人成长也更快了。同时，看到老板的经历，也让我思考起了未来职业发展的一些可能性。比如说，老板早年也是程序员，后来自己成立公司创业，先是接外包，后来自研。还有一个商业合伙人也是在我们当中年纪比较大的，就是靠对业务的理解和一张嘴去拉来客户。我想，自己当然也能一辈子打工，但是人还是要追求自我价值的提升和实现吧。从个人角度出发，先学会一些专业技能，能靠它混到钱，饿不死；然后随着阅历的提升，对社会和人的复杂性有了经验，再加上一点观察力，能说会道的能力上来，就可以拥有自己的事业了。上面的表述并不是说一定要具备这些东西才能创业，也许一开始只具备了一部分，在后续过程中逐渐锻炼出来。但是，无论如何，要有意识地锻炼这些方面，而不是只顾着在专业技能领域里面埋头苦干。我对自己的认知是，除了专业技能外的品质都有很大的进步空间。

- 4.20

今天总结一个很有用的技巧：如果要学习一款软件，且软件的安装比较繁琐，可以 Google "name + playground"，在线环境运行。例子有 typescript、k8s 等。

## 2024.5

- 5.1

五一假期第一天，不想出门，算下来正好到上海一个月了，浅浅总结一下。

首先是工作内容和以前有很大的区别。上份工作我的岗位应该是数据分析师，70%以上的时间在写 raw sql，我好几次提出分担一些写代码的需求，竟然都被拒绝了。平时不仅要完成需求，还要应付客户，被拉到十几个群，每天接受消息轰炸。最重要的是，到后期就感受到自己基本没有成长了，sql 各种花样都见识了（leetcode 刷 sql 题能靠自己写出来中等难度），又没有开发需求，挣钱少，于是就润了。到了这个公司，我目前负责的项目是属于算法实现层面，相对内部，故无需直接面对客户。甚至因为这个属性，都很少打开浏览器 F12 窗口调试。也是因为这个，有一个小小的隐患，就是自己似乎离通常后端所谓的“高薪技术栈”远，比如说高并发、高可用。

不妨现在“吹”一下我负责的项目，以便日后之需。首先它有两种数据库实体，一种是储存 schema（表元数据）的 mongodb，另一种是存实际数据的 OLAP，如 clickhouse（可能应客户需要改为其他数据库）。其次，代码主要有两个作用，对系统外 NL2logicform 模块传入的 LF 进行扩写，使其能转变为正确的 sql。另一个作用是把 LF 转化为 sql/url/api。只有 mongodb 上才会有针对 schema 的全量 crud，clickhouse 只有查询数据。要说这个月哪些方面收获较多，应该有 vscode 工具的熟练度、定位 bug 的能力、JS/TS 的了解程度。因为代码本身质量和完成度还是挺高的，加上老板本人的帮助，理解和修改起来难度不是很大。我在看到这个项目时，也在感叹自己何时也能具备这种从 0-1 的能力。

下一步的学习方向，从微观一点的层面来看，是 TS、Java、Clickhouse 等语言和工具的学习，宏观一点呢，往全栈开发方向走走或许是一个选择。
